**mumei は Claude Code のための品質強制ハーネスです。**

ビルドが緑になった。テストは通った。

……本当に「通った」のでしょうか。それとも、エージェントがテストの方を弱めたのでしょうか。

CI はこの二つを区別してくれません。mumei はここを見ます。

- テストを `HEAD` の内容で実行し直す
- 書き換えさせてはいけないテストは凍結する
- 差分そのものを読んで、赤を緑に変える細工を探す

消されたテスト。握り潰された型エラー。こっそり足された `continue-on-error`。配布物から落ちたファイル。こういうものです。

検査はすべてフックとして OS の境界に置いてあります。エージェントの意図は信頼できない入力として扱います。品質基準はプロンプトでお願いするのではなく、強制します。

ただし、すべてのゲートが同じ強さではありません。ここは正直に書きます。

**強いゲート**は「自分で測り直す」ものです。テストを `HEAD` で実行し直す、スキャナを実際に動かす、`git archive` の中身を実際に開く。これらはすり抜けようとするエージェントにも通用します。

**弱いのは**、mumei が自分の進捗として書き出すファイルです。これは記録であって、防御ではありません。

どちらがどちらかは [何が成立し、何が成立しないか](#何が成立し何が成立しないか) に全部書いてあります。

_名前のない執事。静かに仕え、手柄を取らず、一線を守ります。「それはいたしかねます」。_

## 背景

コーディングエージェントが「書く」工程を引き受けるほど、ボトルネックも移動します。

生み出すことではなく、レビューし、検証し、「何をもって完了とするか」を判断することがボトルネックになります。

mumei はこの変化のために作られました。検査を OS の境界に固定してあるので、コードが誰にも読みきれない速さで届いても効き続けます。([When AI Builds Itself](https://www.anthropic.com/institute/recursive-self-improvement))

この変化には _ループエンジニアリング_ という名前が付きました。エージェントを無人で走らせるループを設計することです。ただし、無人で回るループはミスも無人で量産します。mumei が守るのは、その境界です。

## なぜ mumei なのか

`CLAUDE.md` のルール、システムプロンプト、「先にテストを実行して」。これらはすべてお願いです。能力の高いエージェントは、プレッシャーがかかればお願いを迂回します。

mumei は、守りたい基準をプロンプトから OS の境界へ移します。フックが、プロジェクトを変更する操作（編集・コミット・プッシュ・フェーズ遷移）を検査し、ルールを破るものを拒否します。

決定論的な検出器、複数の視点からのレビュー、証拠を要求する裁定。ここまではもう当たり前で、良いツールがいくつもやっています。

しかし「そのビルド、緑になるよう仕向けられていませんか？」を見ているツールは、他にありません。公式の `/code-review` も、テストカバレッジは対象外だと明言しています。

そこが mumei の持ち場です。

mumei がお願いではなく強制する 4 つ:

- **チャットではなく、ハーネス。**

  フェーズ、Wave、コミット、プッシュ、レビュー工程。すべてフックが決定論的に駆動します。エージェントがプロンプトで切り抜けることはできません。

  抜け道は `MUMEI_BYPASS=1` の 1 つだけです。これは**あなたが**立てる環境変数であり、**エージェントには立てさせません**。Claude Code の設定ファイルに書き込もうとすれば拒否しますし、有効な状態で始まったセッションは開口一番それを白状します。

- **エージェントが黙って弱められないテスト。**

  コミット時、テストは汚れていない `HEAD` のワークツリーで実行し直します。手元に仕込んだ細工は効きません。golden ファイルも、読む前に `HEAD` の内容へ戻します。

  プロパティテストは実装を見せずに書かせます（渡すのは仕様と関数シグネチャだけ）。書いたあとは凍結するので、都合よく調整することもできません。

  そして差分そのものを読み、赤を緑に変える細工を探します。

- **仕様駆動開発（使いたい場合のみ）。** 仕様を生成するツールは数多くありますが、mumei はエージェントに「その仕様どおりに作らせる」ところまでやります。要件 → 設計 → タスク（それぞれ独立にレビュー）→ 一度だけの承認 → Wave 単位の実装、という流れです。フェーズを飛ばしたり、壊れた Wave をコミットしたりすることはできません。

  ただし、**仕様駆動開発は必須ではありません**。mumei には入口が 3 つあります。`/mumei:review` は、同じレビューエンジンと同じ検出器を、いまの差分にそのまま当てます。`.mumei` ディレクトリも作りませんし、状態も残しません。plan 方式は Claude Code 標準の plan モードをそのまま使います。**今のやり方を変えずに、強制の仕組みだけを追加できます。**

- **勘ではなく、根拠に基づくレビュー。** まず決定論的な検出器（CVE・シークレット・型・テスト・SAST）を走らせます。その結果が、続くレビューの根拠になります。レビューは複数の視点で行います（セキュリティ視点と批判的視点を、それぞれまっさらな文脈で動かします）。さらに、指摘ごとに別のエージェントが検証し、根拠の薄いものは「参考」に格下げします。誤検知でマージが止まることはありません。判定はプッシュをせき止めますが、同時に「人間がまだ確認すべき箇所」を必ず示します。

## インストール

コミュニティマーケットプレイスから。Claude Code で:

```text
/plugin marketplace add anthropics/claude-plugins-community
/plugin install mumei@claude-community
/reload-plugins
```

最新版（main 追従）を使う場合は自前のマーケットプレイスから:

```text
/plugin marketplace add iroh4-labs/mumei
/plugin install mumei@mumei
/reload-plugins
```

インストール後、プロジェクトごとに一度だけセットアップを走らせます:

```text
/mumei:kindle
```

アンインストール: `/plugin uninstall mumei@claude-community`（自前マーケットプレイスから入れた場合は `mumei@mumei`。プロジェクト内の `.mumei/` はそのまま残ります）。

前提ツール: レビューフェーズの検出器のために `semgrep` と `osv-scanner` が必要です。インストール手順は [docs/getting-started.ja.md → 前提ツール](./docs/getting-started.ja.md#前提ツール) を参照。

## mumei が強制すること

上の 4 つを、もう少し詳しく:

- **プロンプトではなく、ハーネス** — フェーズ / Wave / コミット / プッシュの各ゲートは、ツール呼び出しの段階で強制されます。エージェントがプロンプトで回避することはできません。
- **状態の保護** — `.mumei/` の状態、レビュー判定、レビュアーが実際に走った証跡は、エージェントには書けません。Edit/Write だけでなく、Bash 経由の書き込みも塞いであります（ルール S1/S2、M1/M2）。ただし完全ではありません。どこまで効くかは [何が成立し、何が成立しないか](#何が成立し何が成立しないか) に書いてあります。
- **テストの細工を見つける** — ビルドが緑でも、「緑になるよう細工されていないか」だけは分かりません。mumei はテストを `HEAD` の内容で実行し直し、実装を見ずに書いたプロパティテストを凍結します。テストを消した差分、型エラーを握り潰した差分、配布物からファイルを落とした差分は、そのまま報告します。
- **誤ってブロックしないゲート** — CVE・シークレット・型エラー・テスト失敗は、判定を `MAJOR_ISSUES` に固定します。ノイズの多い SAST は裁定ゲートを通し、確証が取れたときだけブロックするので、誤検知でマージを誤って止めることはありません。検出ツールが入っていない場合は、致命扱いにせず警告して飛ばします。
- **改ざんできない検証** — コミット時に、テストを汚れていない `HEAD` のワークツリーで再実行します。コミットしていない細工（仕込んだ `conftest.py`、差し替えたレポーター、書き換えたバイトコード）では、合格を偽装できません。
- **エージェントが細工できないテスト** — 不変条件のプロパティテストを、実装を見せずに仕様と関数シグネチャだけから生成し、凍結します。欠陥のある実装に合わせてテストを調整することはできません（AC〔受け入れ基準〕単位で任意に有効化）。
- **多視点のレビュー** — 要件 / 設計 / タスクのレビュアーがまっさらな文脈で独立に走り、続いてセキュリティ視点と批判的視点が差分をレビューします（モデルを入れ替えるのではなく、文脈を非対称にする方式）。さらに指摘ごとの検証担当が、根拠の薄い指摘を「参考」扱いに格下げします。
- **限界を正直に明示** — どの判定にも盲点の但し書きが付き、「人間が手で確認すべき箇所」を明示します。mumei は人間のレビューを不要にするとは主張しません。
- **Wave 単位のコミット** — 1 Wave = 1 コミット。フックが差分を各タスクの `_Files:_` と突き合わせ、実装の差分がないのに `[x]` を付ける「見せかけの完了」を止めます。
- **署名と来歴付きのリリース** — Sigstore キーレス署名、SLSA Level 3、CycloneDX SBOM。詳細は [docs/getting-started.ja.md → Security & supply chain](./docs/getting-started.ja.md#security--supply-chain) を参照。
- **名前のない執事としての姿勢** — mumei は静かに仕え、手柄を取りません。有効化するまで副作用はゼロ（`.mumei/current` がなければフックはすべて何もしません）、余計な発話なし、判定は事実の形式、テレメトリなし。

> より詳しい仕組みは **[ARCHITECTURE.md](./ARCHITECTURE.md)** にあります。hook ID の一覧、検出器の階層、実装を見せずに書くプロパティテスト、レビューの強化、残余リスクの明示、機能をまたいだ指摘台帳、curator が管理するレビュアーの記憶などです。

## 何が成立し、何が成立しないか

ゲートの強さは 2 種類に分かれます。分かれ目は「ローカルか CI か」ではありません。**そのゲートが自分で測り直すか、mumei が書いたファイルを信じるか**です。

### 測り直すゲート（強い）

エージェントの申告を一切あてにしません。

- **テストは `HEAD` から実行し直す。** コミット時、汚れていないワークツリーを作ってそこでテストを走らせます。手元に仕込んだ `conftest.py` や差し替えたレポーターは効きません。golden ファイルも `HEAD` の内容に戻してから使います。tool gate と Stage 0 の検出器は実際にコマンドを起動します。
- **配布物と差分は実物を見る。** `git archive` を実際に叩いて中身を確認し（distribution-shape）、差分そのものを読んでテストの削除や型エラーの握り潰しを探します（gate-weakening）。
- **偽装するにはコミットするしかない。** これらを騙すには細工したファイルをコミットする必要があり、それは差分に残ります。つまり人の目に触れます。

### GitHub 側のゲート（強い、ただし設定が要る）

ブランチ保護（作者以外の code owner による承認、`enforce_admins`、`require_last_push_approval`）は、エージェントの手が届かない唯一の場所です。

これは「念のための多重防御」ではなく、**必須**です。CI の必須チェックは、リポジトリ内のスクリプト（`scripts/lint-all.sh` や `tests/`）を実行しているだけだからです。`Contents: write` のトークンがあれば、lint スクリプトを `exit 0` に書き換えて必須チェックを緑にできます。リポジトリ内の仕組みでこれは塞げません。差分を見た人間だけが止められます。

### mumei が自分で書くファイル（弱い）

`state.json`、タスクのチェックボックス、レビュー判定、`cost-log.jsonl`。これらは測定結果ではなく **mumei の自己申告**です。

フックは、これらへの Edit/Write と、よくある Bash の書き込み経路を拒否します（ルール S1/S2、M1/M2）。おかげで書き換えは「うっかり」ではなく「明らかに変な操作」になります。ただし塞ぎ切れてはいません。Bash のガードはコマンド文字列を読むだけなので、書き込み先を隠したコマンドは通ります。

なので、これらは**記録として扱ってください。壁ではありません**。

### 前提条件（コードでは解決できない）

mumei の「防ぐ」機能は、**エージェントがあなたの認証情報のない場所で動いている**ことを前提にしています。

あなたのマシンで、あなたの `gh` を使えるエージェントは、承認ゲートを迂回しません。**満たしてしまいます**。あなたの認証情報による承認は、GitHub から見ればあなたの承認だからです。

この分離がないと、mumei は「防ぐ」から「後から追える」に格下げされます。そのとき残るのが、最初に挙げた「測り直すゲート」です。だからそこは削りません。

## 研究に基づく設計

mumei の強制モデルは思いつきではなく、エージェントの信頼性とレビュー精度に関する近年の研究から導かれています（各自で参照できるよう arXiv ID を併記します）:

- 能力の高いエージェントは、プロンプト上のルールを選択的に無視します。だから強制は、固い境界（mumei のフック）に置く必要があります。("Formal Policy Enforcement for Real-World Agentic Systems", arXiv 2602.16708; "Willful Disobedience", arXiv 2603.23806)
- エージェントは自分の誤りの大半を見落とします。だからレビュアーはまっさらな文脈で走らせ、自己レビューはさせません。("Self-Correction Bench", arXiv 2507.02778)
- 生の SAST はノイズが多いのですが、構造化された検出結果を LLM に裁定させると精度が大きく上がります。mumei の「検出器 → 検証担当」のゲートはこれに沿っています。("ZeroFalse", arXiv 2510.02534)
- 少数の多様な視点は、同質なエージェントを大量に並べるより優れています。だから mumei は多数決の委員会ではなく、文脈を非対称にしたレビュアーを使います。("Understanding Agent Scaling in LLM-Based Multi-Agent Systems via Diversity", arXiv 2602.03794)

これらは設計に影響を与えたものであり、mumei が各論文の結果そのものを主張するわけではありません。そして次節のとおり、人間のレビューを置き換えるとは決して主張しません。

## コマンド

| コマンド                      | 説明                                                                                                                                                                                                                                                                                                                               |
| ----------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `/mumei:kindle`               | プロジェクトごとに一度だけ走らせるセットアップ。`.mumei/` を作り、`CLAUDE.md` への追記内容を差分プレビュー付きで提案します。                                                                                                                                                                                                       |
| `/mumei:glean <feature>`      | 仕様を書き始める前の任意の Q&A ループ（最大 3 ラウンド × 5 問）。出力は `.mumei/scratch/<feature>.md` に保存されます。                                                                                                                                                                                                             |
| `/mumei:compose [feature]`    | 新規機能では方式を選びます（`spec` = フルの仕様駆動開発 / `plan` = Claude の plan モードのラッパー）。既存の機能は自動で再開します。spec 方式: 確認 → 要件 → 設計 → タスク（それぞれ最大 3 回まで自動レビュー）→ 一度きりの承認 → Wave ごと → レビュー。                                                                           |
| `/mumei:peruse`               | plan 方式のレビュー工程。`pending_review=true` の状態で、Stage 0 検出器 + セキュリティレビュアー + 批判的レビュアー + 指摘ごとの検証担当を、現在の差分に対して回します（最後の `TaskCompleted` が `task_created_count` と一致したときに `pending_review` が立ちます）。                                                            |
| `/mumei:review [base] [spec]` | `git diff $(git merge-base <base> HEAD)`（PR にプッシュ済み + 未コミット）を、共有エンジン（検出器 → レビュアー → 裁定ゲート → 安全側に倒す判定）で単発レビューします。mumei の機能定義は不要で、副作用はゼロ（状態 / 台帳 / 記憶 / コミットを一切書きません）。仕様ファイルを渡すと `spec-compliance-reviewer` が有効になります。 |
| `/mumei:shelve <feature>`     | `done` になった機能を `.mumei/archive/<YYYY-MM>/<feature>/` に移します。方式（specs/ か plans/）を自動判定し、`scratch/<feature>.md` も `scratch.md` として一緒に持ち越します。                                                                                                                                                    |
| `/mumei:muse <feature>`       | アーカイブ済み（またはアーカイブ直前）の機能について `muse.md` を生成します。AC 数 / Wave 数 / レビュー反復のパターン / 修正ループの検出 / トークンコスト / キャッシュヒット率 / フック発火の上位を集計します。読み取り専用、ユーザー起動のみ。                                                                                    |
| `/mumei:attest <feature>`     | 信頼性の詳細ビュー — 直近 10 試行の pass^3 と、`reliability-log.jsonl` の最新 10 行の試行テーブル。読み取り専用、ユーザー起動のみ。                                                                                                                                                                                                |
| `/mumei:glance [feature]`     | 1 行の信頼性サマリ（`<feature> \| pass^3: <value-or-N/A> (n=<n>, window=10, k=3)`）。引数なしは `.mumei/current` を読みます。読み取り専用、ユーザー起動のみ。                                                                                                                                                                      |

## `mumei` がやらないこと

- 人間のレビューの代わりにはなりません。レビュー工程は根拠のある指摘を提示し、盲点を明示しますが、最終判断は人間です。ゲートはしますが、正しさを保証するものではありません。
- CI/CD ツールではありません。フックは Claude Code の中でしか動きません。
- コードレビューサービスではありません。レビュアーはあなたの Claude Code 契約でローカルに実行されます。
- SDD アダプターではありません。mumei は独自の仕様フォーマットを持っています。
- マルチツール対応ではありません。Cursor / Codex / Aider はサポート対象外です。物理的な強制レイヤーは Claude Code のフックに固有のものです。
- ストレージシステムではありません。状態はただのファイルです。DB も MCP サーバーもありません。

## 関連ツール

- **ハーネス化されたレビューワークフロー** — Claude のレビュアーを 4 つの観点（正確性 / セキュリティ / 運用性 / 保守性）で駆動する、再利用可能で可搬な GitHub Actions ワークフロー。semgrep + osv-scanner の出力に基づき、バイアスの中和と「限界の正直な表明」を備えます。どんなリポジトリでも `uses:` 1 行で導入できます。**[docs/review-adoption.md](./docs/review-adoption.md)** を参照。プラグイン本体とは独立しています。
